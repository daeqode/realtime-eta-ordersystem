# 실시간 주문 상태 및 ETA 관리 시스템

---

## Page 1. 프로젝트 개요

> 고객은 주문 전에 대기시간을 확인하고, 운영자는 주문 흐름을 실시간으로 관리할 수 있는 픽업 주문 관리 시스템

| 항목 | 내용 |
|---|---|
| 개발 기간 | 2025년 |
| 개발 인원 | 1인 (기획 · 설계 · 개발 전담) |
| 기술 스택 | Node.js · React · PostgreSQL · Socket.IO |

<br>

스타벅스 같은 픽업 매장에서 흔히 겪는 두 가지 문제 — 고객은 언제 나오는지 모른 채 대기하고, 운영자는 예약 주문을 놓치는 상황 — 를 함께 해결하는 시스템입니다.

고객 앱과 관리자 대시보드가 Socket.IO로 연결되어, 주문 생성과 상태 변경이 양쪽 화면에 동시에 반영됩니다.

**주요 기능**

- 매장별 실시간 ETA · 혼잡도 제공 (주문 전 대기 확인)
- 즉시 픽업 / 예약 픽업 선택 및 주문 추적
- 관리자 실시간 주문 대시보드 (Socket.IO)
- 제조 예정 탭 + reminder badge (예약 주문 운영)
- 주문 상태 변경 시 고객 · 관리자 화면 동시 반영

<br>

**담당 역할**

기획 · 설계 · 개발 전 과정을 1인으로 담당했습니다.

- **주문 상태 흐름 설계** — 접수 대기 → 수락 → 제조중 → 준비 완료 → 픽업 완료 전 단계 설계 및 구현
- **ETA 로직 구현** — 접수 시점 대기 상황을 스냅샷으로 저장하고, 매장 전체 혼잡도를 실시간 재계산하는 로직 구현
- **예약 픽업 · 슬롯 관리** — 시간대별 수용 인원 제한, 마감 임박 표시, 관리자 수동 차단 기능 구현
- **관리자 대시보드** — KPI 카드 · 탭 필터 · 원클릭 상태 전환 · 제조 예정 탭 구현
- **실시간 상태 반영** — Socket.IO 룸 기반 이벤트 설계 및 고객 · 관리자 화면 동시 갱신 구현

## Page 2. 문제 정의 + 해결 방향

### 어떤 문제를 해결했는가

**고객 입장**

- 주문 전, 현재 대기가 얼마인지 알 수 없음
- 주문 후, 내 음료가 어디까지 됐는지 몰라 카운터 앞을 떠나지 못함
- 예약 픽업 슬롯이 얼마나 남았는지 보이지 않아 신뢰하기 어려움

**운영자 입장**

- 새 주문이 들어와도 화면을 새로고침하지 않으면 인지가 늦음
- 예약 주문이 일반 주문과 섞여 있어, 픽업 시각이 다가와도 제조 타이밍을 놓치기 쉬움
- 주문 상태가 한눈에 보이지 않아 운영 실수가 발생하기 쉬움

<br>

### 해결 방향

| 문제 | 해결 방향 |
|---|---|
| 대기 현황 불투명 | 매장별 실시간 ETA · 혼잡도 표시 |
| 주문 인지 지연 | Socket.IO 기반 실시간 반영 |
| 예약 주문 누락 | 제조 예정 탭 + 5단계 reminder badge |
| 상태 파악 어려움 | 탭 필터 + KPI 카드 대시보드 |

단순히 기능을 추가하는 것이 아니라, **운영자가 올바른 정보를 올바른 타이밍에 볼 수 있는 흐름**을 설계하는 것을 목표로 했습니다.

## Page 4. 핵심 설계 결정

### ① ETA를 왜 스냅샷으로 저장했는가
*— 일관된 사용자 경험*

처음에는 주문 추적 화면을 열 때마다 현재 대기열 기반으로 ETA를 실시간 계산하는 방식을 고려했습니다. 하지만 주문 직후 새 주문이 쏟아지면 처음 안내받은 준비 시각이 계속 늦춰집니다. 고객 입장에서는 "6분이라고 했는데 왜 자꾸 늘어나냐"는 상황이 되고, 이는 서비스 신뢰를 직접 떨어뜨리는 구조였습니다.

**결정:** 주문 접수 시점의 ETA를 `OrderEta` 테이블에 스냅샷으로 저장합니다. 이후 대기열이 변해도 고객에게 보여주는 준비 시각은 처음 약속한 값 그대로 유지됩니다.

> 정확한 예측보다 약속한 것을 지키는 것이 신뢰를 만든다.

<br>

### ② 실시간 주문 반영 구조
*— 주문 누락 방지 · 운영 안정성*

관리자 대시보드와 고객 추적 화면이 항상 최신 상태를 보여주려면, 주문 생성과 상태 변경이 발생할 때마다 관련된 모든 화면에 동시에 전달되어야 합니다.

**구조:** 주문 생성 또는 상태 변경 시, 두 가지 이벤트가 함께 발생합니다.

- `order:created` / `order:updated` → 해당 주문 추적 화면 갱신
- `eta:updated` → 매장 전체 혼잡도 및 대기 현황 재계산 후 브로드캐스트

이 구조 덕분에 고객 추적 화면 · 관리자 대시보드 · 매장 목록의 혼잡도 정보가 모두 같은 시점에 갱신됩니다.

> 상태 변경 하나가 관련된 모든 화면에 파급되도록 설계함으로써, 운영 중 정보 불일치를 방지합니다.

<br>

### ③ 제조 예정 탭을 왜 만들었는가
*— 예약 주문 운영 문제 해결*

예약 주문을 일반 주문 탭에 함께 두면 어떤 일이 생기는지 먼저 생각했습니다. 오전에 접수된 오후 2시 픽업 예약이 주문 목록 아래쪽에 묻혀 있다가, 픽업 30분 전이 돼도 눈에 띄지 않습니다. 기능이 있어도 운영자가 인지하지 못하면 없는 것과 마찬가지입니다.

**결정:** "예약 + 아직 제조 시작 전"인 주문만 별도 탭으로 분리하고, 픽업 시각까지 남은 시간을 5단계 reminder badge로 표시합니다.

```
지연 위험      → 픽업 시각 초과
제조 시작 권장  → 15분 미만
곧 제조 필요   → 30분 미만
준비 예정      → 2시간 미만
예약 예정      → 그 이상
```

> 대시보드는 정보를 보여주는 것이 아니라, 지금 무엇을 해야 하는지를 알려줘야 합니다.

## Page 5. 트러블슈팅 + 배운 점

### ① 고객 주문이 관리자 대시보드에 반영되지 않던 문제

**상황**
관리자가 대시보드를 열고 있는 상태에서 고객이 주문해도, 새로고침 없이는 새 주문이 나타나지 않는 경우가 간헐적으로 발생했습니다.

**원인**
관리자 화면은 매장 목록을 불러온 뒤 즉시 `join:store` 이벤트를 emit해 해당 매장 룸을 구독합니다. 그런데 이 시점에 소켓 연결이 아직 완료되지 않은 상태면 emit이 유실됩니다. 연결은 됐지만 룸에 합류하지 못한 상태가 되는 것입니다.

**해결**
소켓의 `connect` 이벤트 핸들러 내부에서 `join:store`를 재처리하도록 구조를 변경했습니다. 연결 타이밍과 무관하게, 소켓이 연결될 때마다 현재 선택된 매장의 룸 구독이 항상 보장됩니다.

> 이 문제가 방치됐다면 관리자가 주문을 인지하지 못해 처리가 지연되고, 실시간 반영이라는 핵심 기능이 사실상 무력화됐을 것입니다.

<br>

### ② 예약 주문 누락 위험

**상황**
예약 주문 기능을 완성한 뒤 실제 운영 흐름을 시뮬레이션했습니다. 오전 10시에 들어온 오후 2시 픽업 예약이, 오후 1시 30분이 돼도 관리자 화면에서 눈에 띄지 않아 제조 타이밍을 놓치는 상황이 충분히 발생할 수 있었습니다.

**원인**
예약 주문이 일반 주문 탭에 섞이면, 주문량이 많을수록 예약 건이 아래로 밀립니다. 픽업 시각이 다가와도 UI에서 어떤 시각적 변화가 없어 운영자가 인지할 방법이 없습니다.

**해결**
제조 예정 탭을 신설해 "예약 + 아직 제조 시작 전"인 주문만 별도로 모았습니다.

각 주문 옆에는 픽업 시각까지 남은 시간을 실시간으로 계산해 5단계 badge로 표시합니다. 탭을 열면 색상만 봐도 지금 처리해야 할 것을 즉시 파악할 수 있습니다.

<br>

### ③ 시연 환경에서 예약 슬롯이 보이지 않던 문제

**상황 · 해결**
운영 데이터 기준으로 오늘 날짜 슬롯이 없거나 이미 마감되면 예약 픽업 흐름 전체를 시연할 수 없었습니다. `NODE_ENV` 기반으로 개발 환경에서는 오늘 날짜 슬롯이 자동 생성되도록 처리해, 운영 데이터 상태와 무관하게 기능 흐름을 온전하게 전달할 수 있었습니다.

---

### 배운 점

**기능 구현과 운영 흐름 설계는 다른 문제입니다.**
API가 동작하고 DB에 잘 저장되더라도, 운영자가 올바른 정보를 올바른 타이밍에 보지 못하면 시스템은 제 역할을 하지 못합니다. 이 프로젝트에서 가장 오래 고민한 것은 기술이 아니라 "이 화면을 보는 사람이 다음에 무엇을 해야 하는가"였습니다.

이 경험은 전산 운영 직무와 맞닿아 있다고 생각합니다. 시스템이 기술적으로 동작하는 것과, 현장에서 실제로 쓰이는 것은 다른 문제입니다. 그 간극을 좁히는 것이 전산팀의 역할이라면, 이 프로젝트에서 그 감각을 직접 다뤄본 셈입니다.
